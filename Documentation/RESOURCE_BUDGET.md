# ATmega328P Resource Budget

This is the accounting contract for browser estimates, export preflight, and
Arduino compile acceptance. It targets only the official hardware profile.

## Physical Limits

| Resource | Physical capacity | Project limit |
| --- | ---: | ---: |
| Flash | 32,768 bytes | 30,720 compiled bytes |
| SRAM | 2,048 bytes | 1,536 known static/fixed bytes |
| EEPROM | 1,024 bytes | 1,024 encoded save bytes |

The ATmega328P contains 32 KB of flash, but bootloader size varies. Existing
Arduino Nano compile reports for this project expose 30,720 bytes. Browser
preflight must therefore use 30,720, not assume that all 32,768 bytes are
available.

The SRAM project limit reserves at least 512 bytes for stack, library calls,
interrupts, and temporary locals. This is a minimum reserve, not a measured
maximum stack depth.

EEPROM stores one manual runtime-state save slot opened from the world-map
system menu. There are no automatic saves and no wear leveling. The save slot
stores compact game progress, not the browser project JSON.

## Status Bands

### Flash

- Pass: compiled size or conservative total is at most 24,576 bytes (80%).
- Warning: 24,577 through 27,648 bytes (80-90%).
- Release review: 27,649 through 30,720 bytes. It fits the observed compile
  target but has less than 10% maintenance headroom.
- Block: more than 30,720 compiled or deterministically generated bytes.

A release-review result is not presented as a normal pass. It requires a
recorded Arduino compile and explicit project decision before release.

### SRAM

- Pass: known globals and fixed runtime buffers total at most 1,280 bytes,
  leaving at least 768 bytes.
- Warning: 1,281 through 1,536 bytes.
- Block: known globals and fixed buffers exceed 1,536 bytes, or a compiler
  reports fewer than 512 bytes remaining for locals.

Unknown stack depth is always visible. Browser preflight must not count it as
zero or claim an exact SRAM pass from static data alone.

### EEPROM

- Pass: exact encoded total is at most 768 bytes.
- Warning: 769 through 1,024 bytes.
- Block: more than 1,024 bytes.
- Unknown: slot encoding is incomplete or any slot/category lacks a byte
  count. Unknown EEPROM cost cannot pass final export.

## Measurement Types

- `Exact generated data`: deterministic bytes emitted into arrays, packed
  tables, buffers, or EEPROM records by the exporter.
- `Estimated compiled cost`: browser planning allowance for functions,
  libraries, compiler output, padding, and code not represented by a table.
- `Measured compile`: Arduino IDE program-storage and global-variable totals.
- `Measured hardware`: upload and real-device behavior.

Exact generated-data totals are repeatable for the same project and exporter.
They are not the final sketch size. Compiler optimization, library code,
linking, padding, globals, locals, stack use, and interrupts prevent browser
arithmetic from replacing an Arduino compile.

When measured compile figures are available, they replace the comparable
browser aggregate for pass/fail reporting; measured globals are not added to
the same static buffers a second time.

## Accounting Owners

Every byte belongs to one category. Shared data is charged once to the module
that emits it.

| Category | Owner | Accounting rule |
| --- | --- | --- |
| Engine/display code | Arduino exporter | Estimated compiled allowance until measured; includes SPI display driver, input, state machine, and shared helpers. |
| World data | World exporter | Exact packed palette, room, coordinate, blocking, section, and lookup bytes in PROGMEM; current 3-bit dense baseline is 9 bytes per room. |
| Visual assets | Visual exporter | Exact emitted glyph, frame, frame-reference, and shared lookup bytes after deduplication. |
| Interactions/events | Event exporter | Exact encoded node, edge, trigger, parameter, and lookup bytes. |
| Text | Text exporter | Exact encoded glyph/text bytes, offsets, and lookup tables after the approved text encoding. |
| Heroes | Character exporter | Exact immutable hero definitions; mutable current-battle copies belong to runtime buffers. |
| Enemies | Character exporter | Exact enemy definitions and immutable stats. |
| Encounters | Encounter exporter | Exact group, member, weight, and formation bytes. |
| Items/equipment | Content exporter | Exact definitions, effects, prices, references, and tables. |
| Spells/abilities | Content exporter | Exact definitions, costs, effects, targets, references, and tables. |
| Battle rules | Battle exporter | Exact rule/configuration tables; battle engine functions remain engine code. |
| Animations | Animation exporter | Exact sequences, frame references, duration data, and animation-only tables. Shared frames remain visual assets. |
| Music/SFX | Audio exporter | Exact note/event/table bytes; audio driver functions remain engine code. |
| Runtime buffers | Runtime design | Exact sizes of fixed mutable arrays and state structures plus a visible stack/temporary reserve. |
| Save data | Save exporter | Exact encoded bytes for one runtime-state slot plus header, validity data, and required metadata. |

JSON file size, browser object size, editor undo history, and IndexedDB
recovery size are not Arduino memory costs.

## Unknown And Future Cost

An unimplemented category is shown as `Unknown`, not zero. Reports must show:

- known exact bytes,
- estimated allowances,
- measured compile totals when available,
- unknown categories,
- remaining capacity after known costs.

Any final combined export with an unknown required category is blocked. A
partial subsystem test may proceed when its omitted categories are clearly
named and the sketch is labeled as a test rather than a complete game.

## PROGMEM And SRAM Rules

- Large immutable world, visual, text, content, animation, and audio tables
  belong in PROGMEM.
- Do not copy the full world or asset libraries into SRAM.
- Decode only the active room, actor, menu, animation, or audio state into
  fixed-size buffers.
- Pointers and indexes emitted for PROGMEM access are counted in their owning
  data category.
- Accidental non-PROGMEM immutable arrays are preflight errors when their
  placement is known.
- Arduino `String`, heap allocation, and unbounded runtime arrays remain
  prohibited.

## Minimal RPG Storage Contract

The minimal beta RPG adds real game state. It must be accounted for before the
combined Arduino export is considered release-ready.

### Authored Data In PROGMEM

These are immutable and should be emitted to `PROGMEM`:

- world and section lookup data,
- wall and locked-gate definitions,
- chest positions and loot kinds,
- section enemy assignment,
- normal enemy stats,
- final-boss stats,
- hero base stats and level-up constants,
- sword attack bonus,
- potion heal amount `10`,
- XP and reward values,
- key, loot, level-up, and ending text bytes.

### Fixed Runtime Buffers In SRAM

These are mutable and must stay fixed-size:

- current world/room/cell position,
- current section,
- hero current HP,
- hero level and XP,
- gold,
- potion count,
- sword obtained/equipped bit,
- key bit,
- gate-opened bit if used,
- boss-defeated bit,
- opened chest bitset,
- active battle enemy HP and battle menu state,
- active display/menu/animation buffers already required by the engine.

The implementation must not copy all chests, all texts, all enemies, or the
full world into SRAM. Decode only the active chest, text, enemy, room, menu, or
battle actor when needed.

### EEPROM Save State

The combined Minimal RPG beta currently uses one manual EEPROM runtime-state
slot opened from the world-map system menu. It encodes:

- save version and validity marker,
- current room/cell position,
- hero current HP,
- hero level,
- hero XP,
- gold,
- potion count,
- sword bit,
- key bit,
- gate bit if used,
- boss bit,
- opened chest bitset,
- checksum or small validation byte.

Project JSON, immutable world data, enemy definitions, item definitions,
visual assets, temporary battle state, and animation state are not stored in
EEPROM.

Multiple EEPROM save slots are not part of the first public beta.

## Pins And Timers

| Function | Pin/resource | Status |
| --- | --- | --- |
| MAX7219 CS/LOAD | D10 | Fixed output |
| Hardware SPI MOSI | D11 | Fixed SPI |
| Hardware SPI SCK | D13 | Fixed SPI |
| Direction/menu pot | A4 | Fixed analog input |
| BrightnessPot | A0 | Fixed analog input |
| Confirm/forward | D5 | Fixed `INPUT_PULLUP` |
| Reject/backward | D4 | Fixed `INPUT_PULLUP` |
| Melody output | D3 | Beta audio output |
| Bass output | D9 | Beta audio output |
| Timer0 | `millis()`, `micros()`, `delay()` | Reserved by Arduino core |
| Timer1 | Audio timing | Used by generated audio runtime when music is assigned |
| Timer2 | Unused by beta audio | Reserved for future work |

The fast beta includes a generated audio runtime for imported worldmap and
battle music. D3 is melody output and D9 is bass output. Projects with no
authored music remain valid; assigned beta music must fit the generated audio
runtime and the official D3/D9 wiring.

Brightness uses `analogRead(A0)` mapped from ADC `0..1023` to MAX7219
intensity `0..15`. All three chips receive register `0x0A` only when the
mapped step changes.

## Evidence Required

### Milestone Test

- Deterministic browser report saved or recorded.
- Arduino IDE compile output with maximum and used flash bytes.
- Arduino IDE global-variable output with maximum, used, and remaining SRAM.
- Exact generated-data category totals for the subsystem under test.
- Upload and hardware result when the milestone changes hardware behavior.

### Public Beta Release

- Complete project has no unknown required categories.
- Preflight has no blocking reference, structure, pin, timer, flash, SRAM, or
  EEPROM result.
- Final generated sketch compiles for the official Arduino Nano target.
- Compile figures are recorded and satisfy the status policy above.
- One-slot EEPROM runtime-state encoding has an exact total at or below 1,024
  bytes.
- Reference game uploads and completes its required hardware test.

## Current Baselines

| Build | Flash | Globals | Result |
| --- | ---: | ---: | --- |
| 2 x 2 combined proof | 10,072 / 30,720 | 580 / 2,048 | Uploaded and accepted |
| 26 x 32 compressed combined proof | 17,580 / 30,720 | 473 / 2,048 | Uploaded and accepted |

The 26 x 32 world baseline used 7,488 packed room bytes plus a small global
palette/flags table in PROGMEM. These measurements guide estimates; they do
not guarantee the size of the completed beta engine.
